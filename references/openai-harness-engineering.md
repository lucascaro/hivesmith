# Harness engineering: leveraging Codex in an agent-first world

- **Source:** https://openai.com/index/harness-engineering/
- **Author:** Ryan Lopopolo (Member of Technical Staff, OpenAI). Acknowledgements to Victor Zhu and Zach Brock.
- **Published:** February 11, 2026
- **Category:** Engineering / Codex 2026

## Thesis

Over five months, a small OpenAI team built and shipped an internal-beta product with **0 lines of manually-written code** — every line (application logic, tests, CI, docs, observability, internal tooling) was written by Codex. They estimate ~10x velocity vs. hand-written code. The post argues that when agents do the writing, the engineer's job shifts from authoring code to **designing environments, specifying intent, and building feedback loops** — i.e. "harness engineering."

Headline metrics:
- Started from an empty git repo in late August 2025; initial scaffold (and `AGENTS.md`) generated by Codex CLI on GPT-5.
- Five months later: ~1M lines of code, ~1,500 PRs merged.
- Throughput: 3.5 PRs/engineer/day, increasing as team grew from 3 to 7 engineers.
- Internal daily users + external alpha testers; humans never directly contribute code.

## Core philosophy

> "Humans steer. Agents execute."
>
> "No manually-written code."

When something fails, the question is never "try harder" — it is **"what capability is missing, and how do we make it both legible and enforceable for the agent?"**

PR loop: Codex opens a PR, reviews its own change locally, requests additional agent reviewers (local + cloud), responds to feedback, and iterates until reviewers are satisfied — described as a "Ralph Wiggum Loop." Humans may review PRs but aren't required to; review is increasingly agent-to-agent.

## Key concepts

### 1. Redefining the engineer's role
Engineers work depth-first: decompose goals into building blocks (design, code, review, test), have the agent build each block, then compose blocks into more complex tasks. Early progress was slow not because Codex was incapable — the **environment was underspecified**.

### 2. Application legibility for the agent
Bottleneck shifted from code throughput to **human QA capacity**. Fixes:
- App is bootable per **git worktree** so Codex can launch one instance per change.
- **Chrome DevTools Protocol** wired into the agent runtime; skills for DOM snapshots, screenshots, navigation → Codex reproduces bugs and validates UI fixes directly.
- Per-worktree ephemeral observability stack: logs (queryable via **LogQL**), metrics (**PromQL**), traces. Enables prompts like *"ensure service startup completes in under 800ms"* or *"no span in these four critical user journeys exceeds two seconds."*
- Single Codex runs regularly work on a single task for **6+ hours** (often overnight).

### 3. Repository as system of record — "give the agent a map, not a 1,000-page manual"
The "one big AGENTS.md" approach failed:
- Context is scarce — a giant file crowds out task/code/docs.
- Too much guidance becomes non-guidance.
- It rots instantly.
- It's hard to verify mechanically.

Solution: `AGENTS.md` is a ~100-line **table of contents**, with a structured `docs/` directory as the system of record. Layout:

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/        (index.md, core-beliefs.md, ...)
├── exec-plans/         (active/, completed/, tech-debt-tracker.md)
├── generated/          (db-schema.md)
├── product-specs/
├── references/         (design-system-reference-llms.txt, nixpacks-llms.txt, uv-llms.txt, ...)
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

Plans are first-class artifacts (lightweight ephemeral plans for small changes; checked-in execution plans with progress + decision logs for complex work). Linters/CI validate doc freshness, cross-links, structure. A recurring "**doc-gardening**" agent scans for stale docs and opens fix-up PRs. This enables **progressive disclosure**: start with a small stable entry point, follow pointers as needed.

### 4. Agent legibility as the design goal
> "From the agent's point of view, anything it can't access in-context while running effectively doesn't exist."

Slack threads, Google Docs, tribal knowledge → invisible to the agent. Push everything into versioned, repository-local artifacts. Favor "boring" technologies (composable, stable APIs, well-represented in training data). Sometimes cheaper to have the agent reimplement a subset of functionality than to depend on opaque upstream code (example: a custom `map-with-concurrency` helper instead of a `p-limit`-style package, fully integrated with their OpenTelemetry instrumentation, 100% test coverage).

### 5. Enforce architecture and taste mechanically
Rigid layered architecture per business domain — strictly validated dependency directions:

```
Types → Config → Repo → Service → Runtime → UI
```

Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through a single explicit **Providers** interface; everything else is mechanically disallowed.

- Custom **Codex-generated linters** + structural tests enforce these rules.
- "Taste invariants" enforced statically: structured logging, naming conventions, file-size limits, platform-specific reliability rules.
- Lint error messages are written to **inject remediation instructions into agent context**.
- Agents are required to **parse data shapes at boundaries** but the specific library is not prescribed (model gravitates toward Zod).

> "This is the kind of architecture you usually postpone until you have hundreds of engineers. With coding agents, it's an early prerequisite: the constraints are what allow speed without decay or architectural drift."

Operating model resembles running a large platform org: **enforce boundaries centrally, allow autonomy locally**. Output may not match human stylistic preferences — that's fine if it's correct, maintainable, and legible to future agent runs.

### 6. Throughput changes the merge philosophy
- Minimal blocking merge gates.
- Short-lived PRs.
- Test flakes addressed with re-runs rather than blocking.

> "In a system where agent throughput far exceeds human attention, corrections are cheap, and waiting is expensive."

### 7. What "agent-generated" actually means
Codex produces:
- Product code and tests
- CI configuration and release tooling
- Internal developer tools
- Documentation and design history
- Evaluation harnesses
- Review comments and responses
- Repository-management scripts
- Production dashboard definitions

When the agent struggles, the team fixes it by feeding the missing tool/guardrail/doc back into the repo — **always by having Codex itself write the fix.**

### 8. Increasing levels of autonomy
End-to-end Codex feature flow from a single prompt:
1. Validate current state of the codebase
2. Reproduce a reported bug
3. Record a video demonstrating the failure
4. Implement a fix
5. Validate the fix by driving the application
6. Record a second video demonstrating the resolution
7. Open a pull request
8. Respond to agent and human feedback
9. Detect and remediate build failures
10. Escalate to a human only when judgment is required
11. Merge the change

Caveat: depends heavily on this repo's structure and tooling — does not generalize without similar investment.

### 9. Entropy and garbage collection
Agents replicate existing patterns — including bad ones — so drift is inevitable. Originally cleaned up manually ("AI slop Friday" — 20% of the week). Didn't scale.

Replaced with **golden principles** encoded in the repo, e.g.:
1. Prefer shared utility packages over hand-rolled helpers (centralize invariants).
2. No "YOLO" data probing — validate at boundaries or use typed SDKs so agents never build on guessed shapes.

A recurring set of background Codex tasks scans for deviations, updates quality grades, and opens targeted refactor PRs (most reviewable in <1 minute, automerged). Functions as **garbage collection** for technical debt.

> "Technical debt is like a high-interest loan: it's almost always better to pay it down continuously in small increments than to let it compound and tackle it in painful bursts."

> "Human taste is captured once, then enforced continuously on every line of code."

## Open questions / what they're still learning
- How architectural coherence evolves over years in a fully agent-generated system.
- Where human judgment adds the most leverage and how to encode it so it compounds.
- How the system evolves as models become more capable.

## Closing takeaway

> "Building software still demands discipline, but the discipline shows up more in the scaffolding rather than the code."

The hard problems shift to designing environments, feedback loops, and control systems that let agents reliably build and maintain complex software at scale.
