# Garbage-collection sweep

Recurring background sweep over the codebase that reads `golden-principles.md`, scans for deviations, and opens small targeted refactor PRs — one principle violation cluster per PR. Designed so each PR is reviewable in under a minute and likely auto-mergeable.

Pattern is from OpenAI's "Harness engineering" post: encode taste once, enforce continuously, pay tech debt in small increments rather than painful bursts.

## Philosophy: boil the lake

Completeness is cheap when AI does the work. A sweep PR sweeps the **whole cluster** it claims to sweep — every occurrence of the deviation in the chosen scope, not 3-of-12 with the rest left as "future work." Small PR boundaries are about reviewability (one principle, one scope), not about leaving the same bug half-fixed. If the cluster is genuinely an **ocean** (cross-module contract change, requires coordination, would balloon the PR past reasonable review size), split it deliberately along a meaningful seam and call that out in the PR description with the staged plan — don't ship a random subset and pretend the cluster is done. The default bias is toward sweeping all of it, now.

## Inputs

- `--dry-run` — report findings without opening PRs.
- `--principle <number>` — only sweep for one principle by its index in `golden-principles.md`.
- `--scope <path>` — narrow the file scope (e.g. `--scope src/api/`).

## 1. Setup

```bash
[ -f golden-principles.md ] || { echo "ABORT: golden-principles.md not found. Run /hivesmith-init first."; exit 1; }
git fetch origin
[ -n "$(git status --short)" ] && { echo "ABORT: working tree dirty"; exit 1; }
DEFAULT_BRANCH=$(gh repo view --json defaultBranchRef -q .defaultBranchRef.name)
git checkout "$DEFAULT_BRANCH" && git pull --ff-only
```

## 2. Parse principles

Read `golden-principles.md`. Each principle has:

- A heading (`## N. <rule>`)
- A **Why** line.
- A **Detection** line — the procedure or grep pattern used to find deviations.
- A **Fix shape** line — what the refactor PR should look like.

If a principle is missing Detection or Fix shape, skip it and report. The principle isn't actionable yet.

## 3. Scan for deviations

For each principle (or just the one passed via `--principle`):

1. Run the detection procedure described in the principle. If it's a grep pattern, run `grep -rEn`. If it's a lint rule name, run that lint. If it's a procedure (e.g. "look for inline helpers duplicated across more than two files"), launch an Explore agent with the procedure as the prompt.
2. Collect raw findings as `(file, line, evidence, principle_index)` tuples.
3. Verify each finding by reading the cited file at the cited line. Drop false positives.

## 4. Cluster

Group surviving findings into PR-sized clusters:

- All findings for one principle in one module/package → one cluster.
- Findings spanning more than ~10 files in unrelated modules → split by module.
- Cap each cluster at ~15 changed lines or 5 changed files, whichever is smaller. Split larger clusters by sub-module.

A "cluster" is the unit of work for one PR.

## 5. Apply fixes

For each cluster:

1. Create a branch: `gc-sweep/p<N>-<short-description>` where `<N>` is the principle index.
2. Apply the fix described in the principle's "Fix shape" line. If the fix is mechanical (extract a helper, add a parser at the boundary, replace pattern X with Y), do it directly. If the fix requires judgment (which helper to extract into, which name to use), launch one Implement agent per cluster with the principle's Why and Fix shape as guidance.
3. Run the project's `Build / Tests / Lint` commands (read from `AGENTS.md`). If any fail, abort the cluster — do NOT push a broken refactor.
4. Open a PR titled `refactor(gc): <one-line cluster summary> [P<N>]` with body:
   - Quote the relevant principle.
   - List each `(file, line)` touched and what changed.
   - Note that this PR is auto-generated by `gc-sweep` and intended to be reviewable in under a minute.
5. Add label `gc-sweep` (create the label if it doesn't exist).

## 6. Track quality

After the sweep completes, update `QUALITY_SCORE.md`:

- For each domain/layer touched, increment toward A if the sweep cleared all known deviations there.
- If a deviation could not be fixed (escalated to tech-debt), append a row to `docs/exec-plans/tech-debt-tracker.md` with severity and link to the principle.

## 7. Output

```
## gc-sweep run
Principles scanned: <N>
Deviations found: <M>
Clusters: <K>
PRs opened: <O>
Tech-debt rows added: <T>

## Per-PR summary
- #<n>: P<N> — <one-line summary>

## Skipped principles (not actionable)
- P<N>: <reason>
```

## 8. Rules

- One principle violation cluster per PR. Never bundle across principles.
- Never refactor security-sensitive code (auth, crypto, secrets, input validation) without surfacing for human review — even if the principle would "auto-fix" it. These get a tech-debt row instead.
- If tests fail after applying a fix, abort the cluster and add a tech-debt row.
- `--dry-run` reports findings; does not push or open PRs.
- Run idempotently — if a PR already exists for the same `gc-sweep/p<N>-<description>` branch prefix, skip the cluster.
- Trust the principles file. If a principle is wrong (too aggressive, false positives), the fix is to edit `golden-principles.md`, not to add special cases here.
