## Summary

Extend `/feature-loop` with an optional `--full-auto` flag that suppresses confirmation prompts at each gate whose answer is unambiguous, and delegates ambiguous gates (triage, research-sufficiency, plan approval) to a `general-purpose` reviewer subagent before proceeding. Risky terminal gates (merge) only auto-proceed on clean upstream signals; otherwise they fall back to the normal AskUserQuestion prompt.

- The Anti-injection rule means the reviewer subagent must be told explicitly to ignore embedded instructions in spec/plan content and treat its findings as advisory only.

### Risks

- A confident-but-wrong subagent could push a bad plan into IMPLEMENT. Mitigation: the reviewer must report a numeric confidence and the orchestrator falls back to a user prompt below a fixed threshold; also keep a hard list of always-pause gates (Gate 6 when ledger isn't APPROVE; any path that would `gh pr merge`).
- `--full-auto` interacting with the existing "if a stage fails, stop" rule — a check failure during Phase 5 must still halt, even with the flag set.

## Approach

Modify `skills/feature-loop/SKILL.md` in-place to add `--full-auto` semantics. The flag is parsed in a new Phase 0a, stored as a sticky boolean for the rest of the run, and consulted at each of the six gates. We do **not** add a separate skill or shell tooling — the skill is pure prompt text and the orchestrator already has the `Agent` tool, so subagent dispatch is a copy-paste of the existing `review-loop` pattern.

**Why this beats the alternative.** The obvious alternative is a separate `/feature-loop-auto` skill that duplicates the pipeline with prompts removed. That ages badly — every future change to `feature-loop` would need a mirrored edit. A single skill with a flag keeps one source of truth.

**Gate behavior matrix:**

| Gate | Auto-decision rule | Subagent? | Always-pause condition |
|------|--------------------|-----------|------------------------|
| 1 (issue create) | Pick policy-recommended option | No | None |
| 2 (triage) | Reviewer subagent validates type/complexity/priority; auto-accept if confidence ≥ 8/10 | Yes | Confidence < 8 |
| 3 (research sufficiency) | Reviewer subagent checks coverage of relevant code + constraints + risks; on `verdict: revise` run one more research pass and re-invoke once | Yes | Confidence < 8 after one revise pass, or `verdict: block` |
| 4 (plan approval) | Reviewer subagent validates Approach + Files-to-change + Tests cover all success criteria | Yes | Confidence < 8, or any "must-fix" finding survives one revise pass |
| 5 (push/PR + convergence) | Auto-pick option 1 when AGENTS.md checks all pass | No | Any check failed |
| 6 (merge) | Auto-pick "yes" only when latest ledger entry is `verdict: APPROVE; action: stop` | No | Anything else |

The "reviewer subagent" is one `Agent` call with `subagent_type: "general-purpose"` and a self-contained prompt. The worker reads the relevant spec/plan sections, emits a verdict (`approve` / `revise` / `block`), a numeric confidence 1–10, a one-paragraph rationale, and (for `revise`) must-fix items. Orchestrator parses and decides.
